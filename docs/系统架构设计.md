# 党员发展材料生成系统 - 系统架构设计

## 一、系统概述

### 1.1 系统定位
本地单机运行的桌面应用程序，用于规范化、自动化生成党员发展过程中所需的多份 Word 表格材料。

### 1.2 核心特性
- **完全离线运行**：不联网、不集中存储数据
- **数据本地化**：所有数据仅保存在本地
- **三级形态管理**：开发者态 → 管理员态 → 成员态
- **数据权限分离**：程序数据、管理员数据、成员数据严格区分
- **模板化生成**：基于 Word 模板自动生成合规文档

## 二、系统形态设计

### 2.1 形态 0：开发者原始态
**特征**：
- 无任何党务数据
- 包含程序逻辑和已变量化的 Word 模板
- 模板中所有待填项使用占位符（如 `{{姓名}}`、`{{支部名称}}`）

**存储内容**：
- 程序代码和资源文件
- Word 模板文件（`.docx`）
- 模板字段映射配置文件（`templates_config.json`）
- 字段定义文件（`fields_definition.json`）

### 2.2 形态 1：支部管理员配置态
**特征**：
- 由党支部负责人使用
- 可配置支部级、组织级信息
- 配置完成后锁定，成员端只读

**可配置内容**：
- 支部名称
- 上级党委信息（名称、负责人等）
- 固定日期格式
- 多个表格共用的公共信息
- 系统设置（导出路径、默认格式等）

**存储位置**：
- `config/admin_config.json`（管理员配置数据）

### 2.3 形态 2：成员使用态
**特征**：
- 成员基于形态 1 的软件填写个人信息
- 无法修改管理员配置
- 只能填写自己的信息
- 可导出 Word 文档

**存储位置**：
- `data/member_info.json`（成员个人数据）

### 4.2 模块职责说明

#### 4.2.1 UI Layer（界面层）
- **首页模块（MemberHomePage）**
  - 显示基本信息填写表单
  - 区分管理员配置字段（只读）和成员填写字段（可编辑）
  - 自动保存成员输入

- **模板填写页模块（TemplatePage）**
  - 每个模板对应一个独立页面
  - 自动引用首页基础信息
  - 显示模板特有字段
  - 避免重复填写

- **导出模块（ExportModule）**
  - 选择要导出的模板
  - 预览生成结果
  - 执行 Word 文档生成

#### 4.2.2 Business Layer（业务层）
- **数据管理模块（DataManager）**
  - 统一管理三类数据的读写
  - 数据验证和校验
  - 数据合并逻辑（基础信息 + 模板数据）

- **权限控制模块（PermissionController）**
  - 判断当前运行形态（开发者态/管理员态/成员态）
  - 控制字段的可编辑性
  - 管理形态切换逻辑

- **模板引擎（TemplateEngine）**
  - 解析 Word 模板中的占位符
  - 执行变量替换
  - 生成最终 Word 文档

#### 4.2.3 Data Layer（数据层）
- **配置数据管理（ConfigDataManager）**
  - 读取/写入管理员配置
  - 提供配置数据的只读接口

- **成员数据管理（memberDataManager）**
  - 读取/写入成员个人数据
  - 数据持久化

- **模板管理（TemplateManager）**
  - 加载模板文件
  - 解析模板配置
  - 管理模板字段映射

#### 4.2.4 File Layer（文件层）
- **JSON存储（JSONStorage）**
  - 本地 JSON 文件读写
  - 数据序列化/反序列化

- **Word生成（WordGenerator）**
  - 使用 docx 库操作 Word 文档
  - 占位符查找和替换
  - 格式保持

- **文件IO（FileIO）**
  - 文件路径管理
  - 文件读写操作

## 六、Word 模板处理方案

### 6.1 模板占位符规范
**格式**：`{{变量名}}`

**示例**：
- `{{姓名}}`
- `{{支部名称}}`
- `{{出生年月}}`
- `{{入党时间}}`

### 6.2 变量替换流程
```
1. 加载 Word 模板文件
   ↓
2. 解析文档结构（段落、表格、页眉页脚）
   ↓
3. 查找所有 {{变量名}} 占位符
   ↓
4. 从数据源获取变量值：
   - 管理员配置数据
   - 成员基础信息
   - 模板特有数据
   ↓
5. 替换占位符为实际值
   ↓
6. 保持原有格式（字体、字号、对齐等）
   ↓
7. 保存为新 Word 文档
```

### 6.3 技术实现方案

#### 方案 A：python-docx（推荐）
**优点**：
- 纯 Python 实现，跨平台
- 支持完整的 Word 文档操作
- 可保持格式

**实现要点**：
```python
from docx import Document
import re

def replace_placeholders(template_path, data, output_path):
    doc = Document(template_path)
    
    # 替换段落中的占位符
    for paragraph in doc.paragraphs:
        for key, value in data.items():
            placeholder = f"{{{{{key}}}}}"
            if placeholder in paragraph.text:
                paragraph.text = paragraph.text.replace(placeholder, str(value))
    
    # 替换表格中的占位符
    for table in doc.tables:
        for row in table.rows:
            for cell in row.cells:
                for key, value in data.items():
                    placeholder = f"{{{{{key}}}}}"
                    if placeholder in cell.text:
                        cell.text = cell.text.replace(placeholder, str(value))
    
    doc.save(output_path)
```

#### 方案 B：docxtpl（基于 python-docx）
**优点**：
- 专门用于模板替换
- 支持 Jinja2 语法
- 更强大的模板功能

**实现要点**：
```python
from docxtpl import DocxTemplate

def generate_document(template_path, data, output_path):
    doc = DocxTemplate(template_path)
    doc.render(data)
    doc.save(output_path)
```

## 七、技术栈建议

### 7.1 桌面应用框架选择

#### 方案 A：Electron + React/Vue（推荐用于快速开发）
**优点**：
- 跨平台（Windows/macOS/Linux）
- 开发效率高
- UI 现代化

**缺点**：
- 体积较大
- 资源占用较高

**技术栈**：
- Electron
- React 或 Vue 3
- TypeScript
- Electron Store（数据持久化）

#### 方案 B：Python + Tkinter/PyQt（推荐用于轻量级）
**优点**：
- 轻量级
- Python 生态丰富
- 跨平台

**技术栈**：
- Python 3.9+
- PyQt6 或 Tkinter
- python-docx（Word 处理）
- json（数据存储）

#### 方案 C：.NET MAUI / WPF（Windows 优先）
**优点**：
- Windows 原生体验
- 性能优秀
- 企业级稳定性

**技术栈**：
- .NET 6+
- WPF 或 MAUI
- DocumentFormat.OpenXml（Word 处理）

## 八、扩展性设计

### 8.1 多省份模板支持
- 模板配置文件支持 `province` 字段
- 根据省份加载对应模板
- 字段定义可差异化配置

### 8.2 新表格接入
- 模板配置采用 JSON 文件，易于扩展
- 新增模板只需：
  1. 添加 Word 模板文件
  2. 在 `templates_config.json` 中注册
  3. 定义字段映射关系

### 8.3 数据备份与恢复
- 支持导出配置数据和成员数据
- 支持从备份文件恢复
- 数据加密存储（可选）

## 九、安全与隐私

### 9.1 数据安全
- 所有数据仅存储在本地
- 不联网、不上传任何数据
- 支持数据加密（可选）

### 9.2 权限控制
- 管理员配置锁定后，成员端无法修改

### 9.3 数据隔离
- 三类数据物理分离存储
- 明确的读写权限控制
